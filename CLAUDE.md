# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Standing rule

**Never modify Verilog/Python source or create a git commit without explicit user approval first.**
Propose the change (what and why), wait for confirmation, then act. This applies even when a fix
seems obvious — a one-line RTL edit changes Vivado place & route determinstically and can silently
invalidate previously-captured calibration data.

## Start-of-session sync check

Before doing any work in a new session, run `git fetch` and compare the local branch against its
remote counterpart (e.g. `git status -sb`, and `git log --oneline HEAD..origin/<branch> --
origin/<branch>..HEAD` for both directions). If there is any difference — local commits not pushed,
remote commits not pulled, or divergence — report it to the user and ask whether to push, pull, or
leave it as-is. Never push or pull without asking first (per the standing rule above).

## File provenance and edit-history conventions

These apply to every file Claude creates or edits in this repo, so that authorship and change
history stay traceable directly in the file (not just in git blame).

**Markdown (`.md`) files** — follow the header style already used in `Markdown/*.md`:
- New file: header block states the creation date and that Claude generated it, e.g.
  `> **작성일** YYYY-MM-DD` / `> **작성** Claude <model name> (Claude Code)`.
- Edit to an existing file: add a revision line to that header block — do not overwrite prior
  revision lines, append to them —
  `> **수정** YYYY-MM-DD HH:MM — <one-line summary of what changed and why>`.

**Verilog (`.v`) and Python (`.py`) files:**
- New file: first line (or first comment block) states it was generated by Claude Code and the
  date, e.g. `// Generated by Claude Code, YYYY-MM-DD` (Verilog) or `# Generated by Claude Code,
  YYYY-MM-DD` (Python).
- Edit to an existing file: leave an inline comment at the edit site — not just a top-of-file
  note — describing what changed, when, and why. Match the existing in-repo style of explaining the
  *reason* for a fix (see the "★ ..." comment blocks in `tdc_test_top.v` / `tdc_histogram.v` for the
  house style: what was wrong, what was changed, why).

## What this project is

A tapped-delay-line Time-to-Digital Converter (TDC) implemented on a Zynq-7000 (Zybo board), built
around a Xilinx `CARRY4` chain. The RTL, the Python calibration/analysis pipeline, and an IEEE
journal paper draft are developed together in this repo — changes to the hardware often require
matching changes to the Python constants and the paper's reported numbers.

There is no Vivado project file (`.xpr`) checked in; this is source-only. Building means creating a
Vivado project, adding the `.v`/`.xdc` files, and generating three IP cores that are referenced by
name but **not present as source** in this repo:

| IP core name | Used in | Purpose |
|---|---|---|
| `clk_wiz_0` | `tdc_test_top.v` | MMCM: fixed 200 MHz + phase-shiftable 200 MHz clock |
| `ila_0` (`universal_ila`) | `tdc_test_top.v` | 6-probe Integrated Logic Analyzer, the only way to get data off-chip |
| `tdc_calib_rom` | `tdc_timestamp_calc.v` | 320×13-bit calibration LUT, latency 2, loaded from a `.coe` |

## Commands

There is no build script — Vivado GUI (or `Tcl` console) drives synthesis/implementation/bitstream.
The repeatable part of the workflow is the Python analysis pipeline, run manually per experiment:

```bash
cd python
python Histogram.py                 # parse Vivado ILA export (iladata.csv) -> tap_histogram_<timestamp>.csv
python Making_COE_Mode_0.py          # histogram -> tdc_calib_mode0_rom.coe (+ tagged copy, SOURCE_TAG)
python Making_COE_linear.py          # uncalibrated linear-ramp COE, for BEFORE/AFTER comparison builds
python DNL_INL_codedensity.py        # cal/val split -> DNL/INL before/after plots
python validate_timestamp.py         # Mode 0 only: timestamp vs loop_cnt linearity (needs true time reference)
```

Each script hard-codes its input CSV filename(s) near the top (`CAL_CSV`, `VAL_CSV`, or `None` for
"most recent `tap_histogram_*.csv`"). Check/update those constants before running — there is no CLI.

Python has no `requirements.txt`; dependencies are `numpy`, `pandas`, `matplotlib`.

## Architecture

### Module hierarchy (data flow)

```
tdc_test_top.v          — mode mux, MMCM phase-shift control, ILA readout scanner, universal_ila
  ├─ mmcm_phase_shifter  (phase_shifter.v)   — drives MMCM dynamic phase shift, 280 steps x 17.857ps
  ├─ tdc_fmcw_core.v                          — the actual TDC: CARRY4 chain + coarse counters + popcount
  ├─ tdc_timestamp_calc.v                     — ROM lookup + pipelined ps-domain subtract -> abs timestamp
  └─ tdc_histogram.v                          — 3-stage RMW BRAM accumulator (code-density histogram)
```

### `OPERATION_MODE` parameter (`tdc_test_top.v`) — three mutually exclusive hit/clock sources

| Mode | Hit source | Clock | Purpose |
|---|---|---|---|
| 0 | `test_hit_sync` (internal, synced) | `clk_200_shifted` | Deterministic calibration via MMCM phase sweep |
| 1 | Ring oscillator (async, random) | `clk_200_fixed` | Statistical calibration (code density), no time reference |
| 2 | `ext_hit_in` (external) | `clk_200_fixed` | Real measurement (LiDAR/FMCW hit) |

Mode 0 has a true time reference (`current_loop_cnt` from the phase shifter), so it's the only mode
where the *timestamp* itself (not just the raw histogram) can be validated against ground truth.
Mode 1 can only validate code density.

### `tdc_fmcw_core.v` — the delay-line core

- 320 taps = 80×`CARRY4` stages, covering one 5 ns clock period.
- Fine index = **popcount of the 320-bit thermometer code** (bubble-immune vs. leading-one detection),
  computed in a 4-stage tree (`stg1_half*` → `stg3_group*` → `sum_fine`).
- Coarse count is **dual-phase**: a 0° counter and a 180°-clocked shadow counter. Near a clock edge
  (`danger_zone`, fine sum outside `[40, 220]`) the 180° value is selected instead, to avoid latching
  a coarse count mid-transition (metastability avoidance, not a timing fix).
- `CYINIT` (not `CIN`) drives stage 0 — this is a general-routing entry point, not the dedicated
  carry-in, and produces an "entry transient" (first few taps have anomalously short delay). `led[2]`
  must never be wired to the hit net — doing so was found to add slew-degrading load to `CYINIT` and
  measurably worsen the entry transient (see the comment block in `tdc_test_top.v` §6).

### `tdc_timestamp_calc.v` — calibration application

`timestamp_ps = ts_coarse * 5000 - calibrated_fine_ps[fine_idx]`, computed in a 5-stage pipeline
(DSP multiply → ROM lookup → borrow-lookahead subtract). The ROM (`tdc_calib_rom`) must have
**latency 2** or the pipeline stage alignment breaks.

### `tdc_histogram.v` — why the RMW is 3 states, not 2

`STATE_RMW_R` (capture `ram_dout_a` only) → `STATE_RMW_A` (`+1`, BRAM out of the path) →
`STATE_RMW_W` (write `count_reg`). Collapsing R+A into one cycle was tried and failed timing
(BRAM `Tcko` + 32-bit adder > 5 ns budget) **and** caused a data hazard (stale `count_reg` from the
previous hit got added to the current address), which was the root cause of a previously-flat-looking
histogram. Do not merge these states without re-verifying both timing and RMW correctness.

### ILA readout addressing — the `probe_read_addr_d1` register

`tdc_bram_512x32`'s port B output is registered (`dout_b <= mem[addr_b]`), i.e. one cycle of read
latency. The ILA samples an address probe and a data probe on the same edge, so the raw sweep address
must be delayed by one cycle (`probe_read_addr_d1`) to stay aligned with `histo_read_data` — otherwise
the whole histogram is shifted by one bin. If you rewire the ILA probes, keep this pairing.

### `universal_ila` has two independent probe groups sharing one ILA core

Probes 0/2/3 (`capture_trigger`/`current_loop_cnt`/`final_timestamp_ps[47:0]`) are for
**timestamp/INL validation**; probes 1/4/5 (`readout_active`/`probe_read_addr_d1`/`histo_read_data`)
are for **code-density histogram capture**. They're captured simultaneously but analyzed by different
Python scripts. If you resize `probe3` (48 bits for the timestamp), the stale comment elsewhere in the
file claiming "probe2 width 32→48" is wrong — check `universal_ila`'s actual probe numbering, not the
commented-out legacy instance below it.

## Calibration methodology (why the Python pipeline is shaped this way)

- **Code density**: tap hit-count histogram ∝ physical bin width. `Making_COE_Mode_0.py` integrates
  the histogram into a cumulative-time LUT; `DNL_INL_codedensity.py` computes DNL/INL from it.
- **cal/val split is mandatory**: the LUT is built from one capture (`CAL_CSV`) and evaluated against
  an independent capture (`VAL_CSV`) of the same build. Never validate a LUT against the data used to
  build it — that measures self-consistency, not calibration quality.
- **Stimulus-agnostic**: the same DNL/INL script works on Mode 0 (MMCM sweep) or Mode 1 (ring
  oscillator) histograms — it only consumes `(Tap_Index, Hit_Count)` pairs.
- `phase_shifter.v`'s `PHASE_STEP = 1000/56 ≈ 17.857 ps` (VCO 1 GHz, 1/56 of the period) is the ground
  truth used by `validate_timestamp.py` for Mode 0's true-time axis. This constant must match whatever
  VCO frequency `clk_wiz_0` is actually configured for.
