## 5. 핵심 발견 — DPS를 code density로 쓴 것이 오용

Korkan 2022 원문 대조 결과:
> *"[DPS의] independence from sufficiently distributed random inputs like code density tests. Instead ... increasing the phase shift between two calibration clocks."*
> *"1024 measurements ... for every 18-ps TI increment, average = result for that TI."*

즉 **DPS의 존재 이유가 "code density처럼 랜덤 균일 입력에 의존하지 않는 것"** 인데,
우리는 DPS를 돌려놓고 **참 시간(loop_cnt)을 버리고 히스토그램만** 썼다. → DPS를 code density
철학으로 오용.

- RO(진짜 랜덤 균일) → code density = 올바름 (0.06)
- DPS(17.857ps 이산 스텝) → code density = 부적절 (0.42, 교차는 더 나쁨)
- **DPS의 정법 = transfer function** (loop_cnt 참 시간 기준). 아직 제대로 안 함.

→ T1 교차검증(DPS-histogram vs RO-histogram)은 사실 "사과 vs 오렌지"였음.
**올바른 교차검증 = RO-code-density vs DPS-transfer-function.**

## 7. 버그 진단 — Mode 0 timestamp 캡처 CDC 미동기화

**증상:** timestamp 캡처가 280스텝 중 **219개만** 잡히고 멈춤. 2회 반복 모두 **동일하게 61개
결손**, 결손 구간 = **loop_cnt 135~205 (위상 ≈ 2400~3660 ps, 주기 중간)**. → 결정론적.

**원인:**
- `current_loop_cnt`는 `clk_200_fixed` 도메인, capture 로직은 `tdc_clk`(Mode 0 = `clk_200_shifted`).
- **다중비트 loop_cnt를 동기화 없이 shifted 클럭으로 샘플** ([tdc_test_top.v:252-269](tdc_test_top.v#L252-L269)).
- loop_cnt가 곧 두 클럭의 위상차를 조절하므로, 중간 위상 대역에서 shifted 에지가 fixed의
  loop_cnt 갱신 에지에 겹침 → 전이값/메타스테빌리티 → `capture_arm` 오동작 → 트리거 결손.

**영향 범위:** Phase 2(timestamp)만. Phase 1(히스토그램)은 loop_cnt 미사용이라 무영향.
`Paper_Roadmap.md`에 기존 미해결로 기록돼 있던 항목이 실측으로 드러난 것.

**해법 후보:** `current_loop_cnt`를 2-FF 동기화 후 비교, 또는 loop_cnt 안정 후 캡처. → **RTL 수정
필요(재빌드·place&route 변경)이므로 승인 전까지 보류.**
